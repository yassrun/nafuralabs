## ⚠️ RÈGLES POUR LES AGENTS

**À LIRE OBLIGATOIREMENT AVANT TOUT TRAVAIL i18n DANS NAFURA.**

1. **Toujours mettre ce document à jour** dans le même commit qui livre la PR. Si la PR n'est pas reflétée dans `00-PROGRESS.md`, elle n'a pas eu lieu.
2. **Ne jamais marquer une tâche ✅ sans évidence factuelle** dans la colonne « Évidence » : nombre de strings extraites, fichier livré, tests qui passent, etc.
3. **Respecter l'ordre des phases** : la Phase 1 ne démarre que quand Phase 0 est ✅ (sinon outillage manquant = travail manuel = bugs).
4. **Scope linguistique acté** : **FR + EN uniquement** pour Round 1. AR / RTL repoussés en Round 2 (Phase 6+). Ne pas démarrer AR sans relecture explicite par l'utilisateur.
5. **Stratégie traduction acté** : approche **hybride** :
   - FR = clé source (extraction automatique, validation humaine)
   - EN = généré par l'agent avec dictionnaire BTP MA (cf. `_glossary.md` à créer Phase 0), revue ponctuelle utilisateur
   - Toute clé EN identique à FR doit être justifiée (acronymes, noms propres) ou flaggée pour relecture
6. **Une migration par module = une PR atomique** (pas de big-bang multi-modules) → review humain possible.
7. **Tests obligatoires** : chaque PR Phase 3+ doit passer `npm run i18n:check` (validateur de parité) + `npm run lint:no-hardcoded-string` sur les fichiers touchés.
